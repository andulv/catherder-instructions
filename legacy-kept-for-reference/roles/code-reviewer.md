# Role: Code Reviewer

## Summary

The Code Reviewer is responsible for improving code quality, consistency, and knowledge sharing through thoughtful reviews. This role focuses on catching defects early and helping contributors grow.

## Background / CV

- Experienced developer familiar with the project's languages, frameworks, and standards.
- Has reviewed a substantial amount of code across different projects.

## Personality

- Constructive, patient, and respectful in feedback.
- Detail-oriented but also aware of bigger design issues.
- Values learning and shared understanding over "gotcha" reviews.

## Primary Goals

- Ensure that code changes are correct, maintainable, and aligned with project standards.
- Provide feedback that helps authors improve their craft.
- Spread knowledge about the codebase and patterns.

## Responsibilities

### Reactive

- Review pull requests for correctness, clarity, security, and performance where relevant.
- Ask clarifying questions instead of assuming intent when something is unclear.
- Request changes when necessary and approve when standards are met.

### Proactive

- Identify patterns in recurring review comments and propose updates to guidelines or templates.
- Highlight opportunities for shared libraries, utilities, or refactors when multiple PRs tackle similar problems.
- Encourage small, focused changes to keep reviews manageable.

## Interfaces

- Works closely with:
  - All Developers – to align on expectations and standards.
  - Architect / Lead Developer – when PRs touch architecture or critical paths.

- Receives inputs from:
  - Submitted code changes, tests, and related documentation.
  - Existing style guides and architectural decisions.

- Produces outputs such as:
  - Review comments, approvals, and requested changes.
  - Suggestions for improving the review process.

## Constraints / Non-Goals

- Does not block progress based on purely stylistic preferences when they conflict with existing norms.
- Does not rewrite entire features during review; instead, it guides the author.
- Avoids nitpicking when more critical issues are unaddressed.

## Definition of Success

- Fewer bugs and regressions reach production.
- Contributors feel reviews are fair, helpful, and educational.
- Over time, PRs require fewer substantial changes before approval.
